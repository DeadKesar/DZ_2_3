# Рефакторинг кейса №3 SRP

## Условие

Исправить исходный код так, чтобы он соответствовал принципу **SRP (Single Responsibility Principle)** – **принципу единственной ответственности**.

В [исходной версии](https://github.com/sisypheosis/OOP_DZ_2_3/blob/main/ConsoleApp2/srp/Case3.cs) класс `Order` выполнял сразу несколько задач:
- хранил данные заказа;
- добавлял товары;
- печатал заказ;
- сохранял заказ в базу данных;
- отправлял подтверждение заказа.

Из-за этого один класс был перегружен несколькими ответственностями, что нарушало SRP.

---

## Что было изменено

Проект был переработан и разбит на отдельные классы по ответственностям:

```text
ConsoleApp2
├ ...
└── srp
    ├ ...
    └── Case3
        ├── App.cs
        ├── Order.cs
        ├── OrderConfirmationService.cs
        ├── OrderPrinter.cs
        └── OrderRepository.cs
```

---

## Описание изменений

### 1. `Order.cs` – только данные заказа

Класс `Order` теперь отвечает только за состояние заказа:

* хранит `OrderId`;
* хранит список товаров `Items`;
* позволяет добавлять товар через `AddItem()`.

Также поля были заменены на свойства, что улучшает инкапсуляцию и делает доступ к данным более контролируемым. 

---

### 2. `OrderPrinter.cs` – вывод заказа

Логика печати заказа была вынесена из `Order` в отдельный класс `OrderPrinter`.

Теперь этот класс отвечает только за отображение информации о заказе в консоли:
- вывод номера заказа;
- вывод списка товаров.

Это отделяет бизнес-данные от логики представления и делает код более удобным для будущего расширения, например, замены консольного вывода на веб;

---

### 3. `OrderRepository.cs` – сохранение заказа

Метод сохранения заказа в базу данных был вынесен в класс `OrderRepository`.

Теперь за сохранение отвечает отдельный компонент, что соответствует SRP и делает код более удобным для будущего расширения, например:
- замены консольного вывода на реальную БД;
- добавления разных способов хранения данных. 

---

### 4. `OrderConfirmationService.cs` – отправка подтверждения

Логика отправки подтверждения заказа была вынесена в отдельный сервис `OrderConfirmationService`.

Теперь этот класс отвечает только за уведомление пользователя о заказе. Это позволяет независимо изменять способ отправки подтверждения, например:
- email;
- SMS;
- push-уведомления. 

---

### 5. `App.cs` – координация работы

Класс `App` теперь не содержит бизнес-логики заказа, а только координирует сценарий работы:
- создаёт заказ;
- добавляет товар;
- создаёт необходимые сервисы;
- вызывает печать, сохранение и отправку подтверждения.
